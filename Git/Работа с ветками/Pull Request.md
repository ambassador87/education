### Pull Request [PR] / Merge Request [MR]
Разница в названии из-за того, что в GitHub это **PR**, а в GitLab это **MR**.
PR - запрос на слияние изменений из одной ветки в другую, из одного репозитория в другой

1. Когда у программистов, которые хотят принять участие в проекте, нет прав на родительский проект,  
*pull request* выполняется через ***FORK*** чужого репозитория.
Функционал PR находится в интерфейсе сервиса репозитория (GitHub, GitLab)

2. Когда два программиста работают над одним проектом в рамках одного репозитория.  
Перед началом, владельцу репозитория нужно добавить другого разработчика в качестве участника  
репозитория. Это делается через интерфейс сервиса репозитория. Указываем аккаунт разработчика, его  
роль и отправляем приглашение. Владелец репозитория будет принимать решение по каждому коммиту.

Новый участник проекта клонирует репозиторий, создает изменения, коммитит в свою ветку и делает пуш  
изменений на сервер. Владалец получает pull request и принимает решение принять его или нет.


Разработчик создает ветку локально
```shell
git checkout -b feature
```
ветка не связана ни с одной веткой на сервере. Чтобы связать, создать upstream, c удаленной веткой:
```shell
# Если ветки еще нет на сервере, но есть локально
git push -u origin имя_ветки
git push --set-upstream origin имя_ветки

# Если ветка уже есть на сервере и локально, но нужно их связать
git branch --set-upstream-to=origin/имя_ветки

# Если ветка уже есть на сервере. Создаст локальную ветку с таким же именем
git checkout --track origin/имя_ветки
```
Что произойдет: Git создаст ветку на сервере и «запомнит», что локальная ветка связана именно с ней.  
В следующий раз тебе достаточно будет просто написать `git push` или `git pull`.

Чтобы убедиться, что всё прошло успешно и ветки видят друг друга:
```shell
git branch -vv
#  master   e52db7e [origin/master] Update Переключение между ветками.md
```
---
Шаг 1: Актуализация (начинаем от main)
Прежде чем создавать что-то новое, нужно убедиться, что у тебя последняя версия кода от коллег.
```shell
git checkout main
git pull origin main
```

Шаг 2: Создание ветки
Создаем новую ветку для задачи и сразу переключаемся на неё.
```shell
git checkout -b feature/add-like-button
```

Шаг 3: Работа и коммиты
Ты пишешь код. Сделал кнопку — закоммитил. Сделал анимацию — закоммитил.
```shell
git add .
git commit -m "Добавлена верстка кнопки лайка"
```

Шаг 4: Первый Пуш и связывание (Upstream)
Теперь отправляем ветку на сервер и устанавливаем связь.
```shell
git push -u origin feature/add-like-button
```
Теперь на GitHub/GitLab появилась ветка с таким же именем.

Шаг 5: Открытие Pull Request
- Заходишь на сайт репозитория.
- Нажимаешь кнопку "Compare & pull request" (она обычно сама выпрыгивает сверху).
-Пишешь описание: что сделал, на что обратить внимание.
-Назначаешь Reviewers (коллег, которые проверят код).

Шаг 6: Правки по результатам Review
Коллега пишет коммент: "Кнопка слишком большая, уменьши". Ты не создаешь новый PR, а просто правишь  
код у себя локально в той же ветке, делаешь коммит и пуш.
Так как связь (-u) уже была установлена, новый пуш сам «долетит» в тот же открытый PR.

Шаг 7: Слияние (Merge)
Когда ревьюер ставит "Approve", а тесты (CI) прошли успешно:- 
- Владелец (или ты) нажимает Merge pull request на сайте.
- Твои коммиты попадают в main на сервере.
- Ветка feature/add-like-button на сервере обычно удаляется кнопкой "Delete branch".

Шаг 8: Уборка дома (локально)
Теперь твой локальный main устарел (там нет твоей кнопки), а ветка feature больше не нужна.
```shell
git checkout main
git pull origin main       # Твоя кнопка «вернулась» к тебе уже в составе мейна
git branch -d feature/add-like-button  # Удаляем локальную ветку за ненадобностью
```

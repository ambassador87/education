При слиянии веток могут возникнуть две ситуации:
1. **Fast Forward Merge** - если после ответвления в главной ветке не было коммитов, главная ветка  
сдвигается вперед, указывая на последний коммит второстепенной ветки.
2. **Merge Commit** - если после ответвления в главной ветке были коммиты, тогда создается 
новый merge-коммит с двумя родителями (main и feature), содержащий в себе изменения из главной 
и второстепенной веток. Указатель main сдвигается на новый merge коммит, указатель feature остается  
на месте.

***ВАЖНО ЗНАТЬ!***  
Перед слиянием своей ветки с master/main локально, нужно актуализировать свою мастер ветку  
с удаленной (*origin/master*), т.е. получить свежие изменения из главного репозитория,  
которые могли добавить другие разработчики.
```shell
# git pull <remote> <branch>
git pull origin/master

# Или то же, но разбитое на два шага
git fetch origin
git merge origin/master
```
***

Чтобы слить свою ветку в мастер, нужно сначала переключиться на мастер и после сделать  
слияние веток:
```shell
# Переключает на мастер ветку
git swtich master

# Делает слияние ветки с мастер веткой
git merge <имя_ветки>
```
В ситуации, когда мы не хотим делать fast forward merge, можно принудительно  
сделать обычный merge commit:
```shell
git merge --no-ff <имя_ветки>
```
Так как при обычном merge commit происходит создание нового коммита в целевой ветке,  
к нему можно добавить комментарий, используя ключ `-m`:
```bash
git merge --no-ff <имя_ветки> -m "Это мой мердж коммит"
```
